# ADR-001: DB-Kafka Consistency via AFTER_COMMIT Event Listener

## Status
Accepted

---

## Context

The order-service must update order status in PostgreSQL and publish a
corresponding event to the `order-status-changed` Kafka topic as part of every
successful status transition. These two operations cannot be wrapped in a single
atomic transaction — PostgreSQL and Kafka do not share a transaction coordinator.

This creates a **split-brain window**: if the application crashes or the Kafka
broker is unavailable between the two operations, one side has the change and the
other does not.

Additional constraints that shaped the decision:

- The internal status-update caller retries the same transition up to **3 times**.
  Both the DB write and the Kafka publish must be idempotent.
- The assignment scope is **6–8 hours** — solutions requiring additional
  infrastructure (CDC connectors, outbox pollers) increase complexity
  disproportionate to the time budget.
- Phantom events (event published for a transaction that rolled back) are **more
  dangerous** than lost events for this domain: a spurious "order confirmed"
  notification is a worse user experience than a delayed one.

---

## Options Considered

### Option 1: Direct Kafka publish inside the `@Transactional` method

Publish to Kafka inline, within the same method that writes to the DB, before
the transaction commits.

```java
@Transactional
public OrderResponse updateStatus(...) {
    order.setStatus(newStatus);
    orderRepo.save(order);
    kafkaTemplate.send(TOPIC, buildMessage(order)); // inline
    return OrderResponse.from(order);
}
```

**Pros:**
- Simplest implementation — no extra infrastructure or wiring.

**Cons:**
- **Phantom events**: if Kafka `send()` succeeds but the DB transaction
  subsequently rolls back (e.g., validation error, constraint violation),
  downstream consumers receive an event for a state change that never
  persisted. This is the more dangerous failure direction.
- **Lost events**: if the DB commits but Kafka is unreachable at publish time,
  the event is permanently dropped with no recovery path.
- Kafka publish latency is added to the transaction hold time, increasing
  lock contention under load.

**Verdict:** Rejected. Phantom events are unacceptable.

---

### Option 2: `@TransactionalEventListener(AFTER_COMMIT)` — **CHOSEN**

Publish a Spring `ApplicationEvent` inside the DB transaction. A
`@TransactionalEventListener` with `phase = TransactionPhase.AFTER_COMMIT`
fires the actual Kafka publish only after the DB transaction has committed
successfully.

```java
// Inside the @Transactional service method:
eventPublisher.publishEvent(
    new OrderStatusChangedEvent(orderId, customerId, previousStatus, newStatus)
);

// Separate listener — fires only after commit:
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void handleStatusChanged(OrderStatusChangedEvent event) {
    kafkaTemplate.send(TOPIC, buildMessage(event));
}
```

**Pros:**
- **No phantom events**: if the DB transaction rolls back for any reason, the
  listener never fires and no event is published.
- No additional infrastructure (no outbox table, no poller, no CDC connector).
- Simple to reason about and test.
- Naturally handles the "internal service retries 3×" requirement: the
  idempotency check (`order.status == newStatus`) short-circuits before
  publishing a duplicate event on replay.

**Cons:**
- **At-most-once delivery**: a JVM crash between the DB commit and the
  `AFTER_COMMIT` listener execution results in a permanently lost event.
  The DB has the new status; Kafka does not.
- **Kafka broker unavailability**: if the broker is unreachable at the moment
  the listener fires, the event is dropped. There is no automatic retry at the
  broker level from this listener.
- The Kafka publish is fully decoupled from the DB transaction — there is no
  mechanism to "undo" the DB change if Kafka publish fails.

**Verdict:** Accepted for assignment scope. See Trade-offs Accepted.

---

### Option 3: Transactional Outbox Pattern

Within the same DB transaction that updates order status, write a record to an
`outbox` table. A separate relay process reads unpublished outbox records and
publishes them to Kafka, then marks them as published.

```sql
-- Added to the same @Transactional as the status update:
INSERT INTO outbox (id, aggregate_id, event_type, payload, created_at)
VALUES (gen_random_uuid(), :orderId, 'ORDER_STATUS_CHANGED', :payload, NOW());
```

The relay can be implemented as:
- A scheduled `@Scheduled` polling job (simple, 1–5s latency).
- A Debezium CDC connector reading the Postgres WAL (sub-100ms latency, no
  polling overhead, but requires Kafka Connect infrastructure).

**Pros:**
- **Fully durable**: the outbox record is written atomically with the DB update.
  A JVM crash after commit leaves the outbox record intact; the relay picks it
  up on recovery.
- **At-least-once delivery**: survives broker outages, JVM crashes, and network
  partitions. The relay retries until Kafka acknowledges the publish.
- **Exactly-once achievable**: with consumer-side `eventId` deduplication (each
  message carries a UUID), at-least-once delivery + idempotent consumers =
  effective exactly-once semantics.
- Fully decouples the application from Kafka availability at request time.

**Cons:**
- Requires an `outbox` table, a relay service/job, and (for Debezium) a Kafka
  Connect cluster with `wal_level = logical` in Postgres config.
- Adds operational complexity: the relay process must be monitored, the outbox
  table must be periodically cleaned up, and relay lag must be alerted on.
- Polling-based relay introduces 1–5s event latency; Debezium reduces this to
  < 100ms but adds infrastructure.

**Verdict:** The correct production solution. Not implemented due to assignment
time constraints. Documented as the target production architecture.

---

## Decision

**Option 2 (`@TransactionalEventListener(AFTER_COMMIT)`)** was chosen for the
scope of this assignment.

The primary decision criteria:

1. **Phantom events are eliminated.** For a food delivery platform, a spurious
   "your order is confirmed" notification is a worse user experience than a
   delayed or missing one. Option 2 is strictly safer than Option 1 in this
   regard.

2. **Complexity is proportionate to scope.** The assignment explicitly permits
   documented trade-offs made under time pressure. Option 3 would require
   significant additional infrastructure (outbox table, relay, Debezium) with
   limited visibility in an 8-hour context.

3. **The failure mode is documented and recoverable.** The at-most-once gap is
   narrow (JVM crash between commit and listener execution) and is fully
   documented in the README failure analysis section. In production, this would
   be closed by migrating to Option 3.

---

## Trade-offs Accepted

| Trade-off | Implication |
|-----------|-------------|
| **At-most-once Kafka delivery** | A JVM crash between DB commit and `AFTER_COMMIT` listener execution permanently loses the event. Downstream consumers do not receive that status change notification. |
| **No broker-level retry** | If Kafka is unavailable when the listener fires, the publish fails silently (error is logged). There is no retry queue or dead-letter mechanism. |
| **Event delivery coupled to JVM liveness** | Unlike the Outbox pattern, event delivery is not independent of the application process. If the application is unhealthy, events stop flowing even if the DB is healthy. |

## Failure Scenario Analysis

**Scenario: JVM crash after DB commit, before `AFTER_COMMIT` listener fires.**

State after crash:
- PostgreSQL: order status updated to new value, `version` incremented,
  `idempotency_keys` record present (for creation events).
- Kafka: no event published to `order-status-changed`.
- Downstream consumers: unaware of the status change.

Recovery path:
- The DB is the source of truth. The order is in the correct state.
- If the triggering client retries the status update, the idempotency check
  (`order.status == newStatus`) returns `200 OK` without re-publishing —
  the event remains lost.
- Downstream systems must reconcile against the DB to detect the gap.

**This is the known, accepted limitation of Option 2.**

In production, the Transactional Outbox (Option 3) closes this gap: the outbox
record was written in the same DB transaction, so the relay publishes the event
on recovery regardless of JVM crash timing.

## Idempotency Design

### Order creation
The `Idempotency-Key` header value is stored in the `idempotency_keys` table
**within the same DB transaction** as the order insert. Both writes commit or
roll back together — atomicity is guaranteed by the transaction boundary.

A concurrent duplicate request (same key, different pod) will fail on the
PostgreSQL unique constraint (`idempotency_key` PK), triggering a
`DataIntegrityViolationException`. The catch block re-reads the stored response
and returns it — the same result as a sequential duplicate.

### Status update (internal service retries)
Before attempting a state transition, the service checks:
```java
if (order.getStatus() == newStatus) {
    return OrderResponse.from(order); // idempotent replay — no event published
}
```
This ensures that up to 3 retries from the internal service:
1. Do not produce duplicate DB writes.
2. Do not publish duplicate Kafka events.
3. Return `200 OK` with the current order state on all replays.

---

## Ordering Guarantee

Kafka events are published using `orderId` as the partition key:
```java
kafkaTemplate.send(TOPIC, orderId.toString(), message);
```
All events for a given order are routed to the same partition, preserving
insertion-order delivery to consumers. A consumer will always see
`PENDING → CONFIRMED → PREPARING` in that order for the same `orderId`.

## EventId and Consumer Deduplication

Every published Kafka message includes a unique `eventId` (UUID):
```json
{
  "eventId": "a4f1c3e2-...",
  "orderId": "...",
  "previousStatus": "PENDING",
  "newStatus": "CONFIRMED",
  "timestamp": "2024-01-15T10:30:00Z"
}
```
Consumers that require exactly-once processing can use `eventId` to deduplicate:
```sql
INSERT INTO processed_events (event_id, processed_at)
VALUES (:eventId, NOW())
ON CONFLICT (event_id) DO NOTHING;
-- affected rows = 0 → duplicate, skip processing
```

## Target Production Architecture

The following architecture would replace Option 2 in a production deployment:

```
Order Service (DB Transaction)
  ├── UPDATE orders SET status = 'CONFIRMED' WHERE id = :id
  └── INSERT INTO outbox (aggregate_id, event_type, payload) VALUES (...)

Debezium CDC Connector
  └── Reads Postgres WAL → publishes to Kafka order-status-changed topic

Kafka Consumers
  └── Deduplicate via eventId → process event
```

Requirements:
- `wal_level = logical` in `postgresql.conf`
- Kafka Connect cluster with `debezium-connector-postgres`
- `outbox` table with `(id, aggregate_id, event_type, payload, published_at, created_at)` schema
- Outbox cleanup job (DELETE WHERE published_at IS NOT NULL AND created_at < NOW() - INTERVAL '7 days')

This provides **at-least-once delivery** that survives any crash scenario, with
consumer-side `eventId` deduplication achieving effective exactly-once semantics.
