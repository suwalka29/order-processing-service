# ADR-001: DB-Kafka Consistency via AFTER_COMMIT Event Listener

## Status
Accepted

## Context
The order-service must atomically update order status in PostgreSQL
and publish an event to Kafka. A naive approach (update DB then
publish to Kafka inline) creates a split-brain window: if the app
crashes between the two operations, one side has the change and
the other does not. The internal status-update caller retries up
to 3 times, meaning idempotency of both the DB write and the event
publish is required.

## Options Considered

### Option 1: Direct publish in @Transactional method
Publish to Kafka inside the same method that writes to DB.
- Pro: simple, minimal code
- Con: if DB commits but Kafka publish fails (broker down, network
  blip), the event is permanently lost with no retry mechanism.
  Conversely if Kafka succeeds but DB rolls back, a phantom event
  is published.

### Option 2: @TransactionalEventListener(AFTER_COMMIT) — CHOSEN
Publish a Spring ApplicationEvent inside the DB transaction.
A @TransactionalEventListener fires the actual Kafka publish only
after the DB transaction has committed successfully.
- Pro: no phantom events (if TX rolls back, listener never fires)
- Pro: simple to implement, no extra infrastructure
- Con: crash between DB commit and listener firing = lost event
- Con: Kafka publish happens outside the transaction, so no
  automatic retry on broker failure

### Option 3: Transactional Outbox Pattern
Write to an outbox table within the same DB transaction. A separate
poller reads unpublished records and publishes them, then marks
as sent.
- Pro: fully durable, survives crashes
- Pro: exactly-once semantics achievable with deduplication
- Con: requires additional infrastructure (poller, outbox table,
  scheduler)
- Con: adds latency (poll interval)

## Decision
Option 2 (AFTER_COMMIT listener) for the scope of this assignment.
This eliminates phantom events (the more dangerous failure mode)
while keeping the implementation simple and explainable.

This approach was chosen to balance correctness and implementation
complexity within the time constraints of the assignment (6–8 hours).
While the Outbox Pattern provides stronger guarantees, it introduces
additional components (outbox table, poller/CDC) that would increase
implementation time and complexity. Given the scope, AFTER_COMMIT
provides a reasonable trade-off while still demonstrating awareness
of distributed consistency challenges.


## Trade-offs Accepted
1. **At-most-once delivery**: a crash between DB commit and Kafka
   publish results in a lost event. Downstream consumers must
   tolerate missing events or reconcile via the DB.
2. **No broker-level retry**: if Kafka is temporarily unavailable
   at publish time, the event is dropped. In production, Option 3
   (Outbox) would be used to provide at-least-once guarantees.


## Failure Scenario
If the application crashes after the database commit but before the
AFTER_COMMIT listener executes, the order state is persisted but the
corresponding Kafka event is never published.

This results in a lost event and downstream systems may miss updates
unless they reconcile state from the database.

## Idempotency Consideration
Each Kafka message includes a unique eventId. This allows downstream
consumers to implement deduplication if retries or duplicate deliveries occur.

## Ordering Guarantee
Events are published using orderId as the Kafka message key.
This ensures that all events for a given order are routed to the same
partition, preserving ordering guarantees per order.

## What I Would Do in Production
Implement the Transactional Outbox Pattern with a Debezium CDC
connector reading the outbox table's WAL changes and publishing
to Kafka. This gives at-least-once delivery and survives any
crash scenario.
